Cache Strategy for Read-heavy Configuration Data

This page explains how I selected a local caching strategy instead of Redis to reduce database load and eliminate unnecessary overhead for frequently read data.

Context

  • Configuration data such as attendance rules, event settings, and operational notices
  • Frequently read but rarely updated
  • Caused unnecessary DB connection usage during peak traffic

Design Decision

  • Applied local caching (Ehcache) since the data changes infrequently and does not require strong consistency
  • Chose Ehcache over Redis to:
    • Eliminate network overhead
    • Minimize operational complexity

Implementation

  • Implemented caching using Spring Cache Abstraction with Ehcache
  • Applied TTL-based eviction policy
  • Provided admin-triggered cache invalidation
    • Cache is refreshed when notices are published or unpublished

Result

  • Eliminated DB access for configuration-related read APIs
  • Stabilized DB connection usage during peak hours
  • Limited cache usage to configuration data only, minimizing data consistency risks
